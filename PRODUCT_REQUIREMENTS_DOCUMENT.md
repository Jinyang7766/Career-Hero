# Career Hero - 产品需求文档（PRD）

## 1. 文档信息
- 文档版本：v2.0（基于当前代码实现评估后重构）
- 适用范围：`Career-Hero` 仓库（前端 `ai-resume-builder` + 后端 `backend`）
- 文档目标：统一产品、研发、测试对“当前已实现能力 + 下一步优化重点”的理解

## 2. 产品定位
Career Hero 是一个面向求职用户的 AI 训练平台，核心闭环为：
1. 结构化简历沉淀
2. 面向 JD 的智能诊断
3. 场景化 AI 面试训练
4. 结果报告与持续迭代

产品不是单点“简历生成器”，而是“诊断 + 训练 + 复盘”的持续提升工具。

## 3. 当前范围定义（In Scope / Out of Scope）

### 3.1 In Scope（当前版本）
- 账号体系：注册、登录、会话状态维护、账号安全入口
- 简历管理：创建、编辑、预览、导出、重命名、删除、多简历列表
- AI 诊断：JD 输入、分析、初步报告、微访谈、最终报告
- AI 面试：场景配置（类型/模式/训练重点/公司/JD）、聊天面试、面试报告
- 积分体系：诊断与面试扣点、失败返还、会员引导
- 状态恢复：跨页面/刷新恢复诊断和面试进度

### 3.2 Out of Scope（当前版本不承诺）
- 外部招聘平台自动投递
- 多语言全链路（简历+面试）生产可用
- 企业招聘端协同（面试官后台、团队管理）

## 4. 用户与核心场景

### 4.1 目标用户
- 应届生：需要模板化引导和面试练习
- 跳槽用户：需要针对目标 JD 快速迭代简历
- 转型用户：需要将既有经历映射到新岗位语言体系

### 4.2 关键场景
1. 用户选择某份简历 -> 输入 JD -> 获得诊断结论 -> 按建议修改
2. 用户进入 AI 面试 -> 配置场景并开始 -> 中断后可继续 -> 完成后生成报告
3. 用户在多份简历间切换时，状态必须“按简历隔离”，不串扰

## 5. 功能需求（按模块）

### 5.1 账号与会话
- 支持邮箱登录/注册
- 未登录访问受限页面时需阻断并提示
- 本地会话缓存按用户隔离，避免不同账号互相污染历史状态

### 5.2 简历生命周期管理
- 创建与编辑：
  - 结构化字段：个人信息、工作、教育、项目、技能、总结
  - 自动保存 + 手动保存并预览
- 列表展示：
  - “上次修改”优先展示 `resume_data.contentUpdatedAt`，回退 `updated_at/created_at`
  - 诊断进度与面试阶段按简历单独计算
- 操作：
  - 预览、重命名、删除、批量删除
  - 历史导出记录展示

### 5.3 AI 诊断流程
- 步骤状态：`jd_input -> analyzing -> report -> micro/chat -> final_report`
- 支持从列表点击简历后恢复到“可恢复的最新诊断步骤”
- 诊断与面试信号必须分离：面试状态不能误导诊断跳转
- 回退逻辑：报告页返回不能出现“需点两次才返回”的假回退

### 5.4 AI 面试流程
- 场景配置项：
  - 面试类型（通用/技术/HR）
  - 面试模式（简单/全面）
  - 自定义训练重点（interviewFocus）
  - 目标公司/岗位（targetCompany）
  - JD 文本（lastJdText）
- 开始面试：
  - 点击“开始面试”并确认后，必须进入聊天页
  - 若恢复会话失败，不得阻断进入聊天
- 锁定规则：
  - 面试进行中（`interview_in_progress/paused`）时，上述场景字段锁定
  - 从聊天页返回设置页，保持锁定与原值
  - 返回“选择简历”再进入同简历，锁定状态与字段值仍保持

### 5.5 报告与导出
- 初步报告：诊断评分、优劣势、建议项
- 面试报告：面试总结、得分、改进建议
- 最终报告：面试后综合输出
- 导出：支持 PDF / 图像导出并记录导出历史

### 5.6 积分与会员
- 扣点动作可配置（诊断、面试 simple/comprehensive、最终报告等）
- 扣点失败需提示并阻断流程
- AI 失败/超时/取消触发返还策略

## 6. 状态与数据约束（关键）

### 6.1 核心数据字段（ResumeData）
- 简历内容：`personalInfo/workExps/educations/projects/skills/summary`
- 内容更新时间：`contentUpdatedAt`
- 诊断相关：`analysisSnapshot/analysisSessionByJd/analysisBindings/latestAnalysisStep`
- 面试相关：`interviewSessions/interviewFocus/targetCompany/lastJdText`

### 6.2 一致性规则
- 同一用户、多份简历状态严格隔离
- 读取简历时，最终以“数据库行 id”为准，禁止被 `resume_data.id` 脏值覆盖
- 任何会话恢复失败都应“降级可用”，不能导致页面卡死或无法跳转

## 7. 非功能需求（NFR）
- 可用性：
  - 核心流程（选简历 -> 开始诊断/面试）无阻断死链
  - 刷新后可恢复最近有效状态
- 性能：
  - 首屏可交互 < 2.5s（常规网络）
  - 关键按钮反馈 < 300ms（loading/禁用态）
- 安全：
  - 接口鉴权与用户数据隔离
  - 敏感信息在 AI 调用前做脱敏处理
- 可观测性：
  - 关键流程日志可定位（会话恢复、步骤跳转、状态持久化）

## 8. 验收标准（重点回归清单）
1. 选择简历后跳转到该简历最新诊断步骤，不回落到错误页面
2. 多简历场景下进度条不串扰
3. 报告页返回单击即可生效
4. 开始面试后可进入聊天页（即使恢复函数异常）
5. 聊天返回后场景字段锁定且不丢值
6. 返回简历列表再进入，锁定状态仍正确
7. 简历“上次修改”展示为 `contentUpdatedAt`

## 9. 技术实现概览
- 前端：React 18 + TypeScript + Vite + Tailwind
- 状态管理：Zustand + 局部 Hook 状态机
- 后端：Flask（AI 接口与业务编排）
- 数据层：Supabase PostgreSQL（含 JSON 字段与业务表）
- 模型与智能能力：Gemini 系列（按场景调用），支持缓存与可选 RAG 增强

## 10. 近期版本建议（Roadmap）
- P0：稳定性与一致性
  - 继续补齐诊断/面试状态恢复单测
  - 增强会话恢复失败的可观测日志与降级策略
- P1：效率与体验
  - 报告页面加载与切换性能优化
  - 面试启动路径的用户提示与确认流程精简
- P2：能力扩展
  - 多语言简历优化
  - 面试历史回放与训练轨迹可视化

## 11. 模型切换成本与毛利评估（Qwen / DeepSeek）

### 11.1 评估口径（固定）
- 评估日期：2026-02-25
- 汇率口径：全部按人民币（¥）计算
- 本产品积分与定价口径（来自当前代码）：
  - 会员月积分：Starter 220 / Plus 800 / Pro 1500 / Ultra 3000
  - 会员月费：Starter ¥9.9 / Plus ¥29.9 / Pro ¥49.9 / Ultra ¥99.9
  - 积分消耗：初步诊断 10、微访谈 20、最终报告 20、面试简单 20、面试全面 50
- 单位积分收入（元/积分）：
  - Starter `9.9/220=0.0450`
  - Plus `29.9/800=0.0374`
  - Pro `49.9/1500=0.0333`
  - Ultra `99.9/3000=0.0333`

### 11.2 模型单价口径（官方）
- Qwen（阿里云百炼，单位换算为元/百万 token）：
  - Qwen-Turbo：输入 `0.3`，输出 `0.6`
  - Qwen-Plus（非思考模式）：输入 `0.8`，输出 `2.0`
- DeepSeek（DeepSeek API，deepseek-chat）：
  - 命中缓存输入 `0.2`，未命中缓存输入 `2.0`，输出 `3.0`
- 本文“保守估算”统一按 DeepSeek 输入未命中缓存计价（`2.0`）。

### 11.3 单次调用 token 假设（用于财务估算，后续建议接入真实埋点替换）

### 11.6 与人工优化简历成本对比（运营视角）
按内部交付成本法估算（建议财务每月校准）：
- 人工优化成本公式：`人工时薪（含管理与社保分摊） × 单份耗时`
- 参考区间：
  - 轻量版：`¥80/小时 × 1.5小时 = ¥120/份`
  - 标准版：`¥150/小时 × 2.0小时 = ¥300/份`
  - 深度版：`¥280/小时 × 2.5小时 = ¥700/份`

与 AI 成本对比（完整诊断闭环 50 积分）：
- Qwen-Turbo：约 `¥0.0184/次`
- Qwen-Plus：约 `¥0.0520/次`
- DeepSeek-Chat（未命中缓存）：约 `¥0.1148/次`

结论：
- 在当前会员定价下，AI 单次边际成本远低于人工交付成本（数量级差异）。
- 产品可将人工服务定位为“高客单价增值服务”，AI 作为规模化主交付路径。
